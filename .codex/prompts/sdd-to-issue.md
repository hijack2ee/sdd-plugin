
# sdd-to-issue

manifest와 requirements를 기반으로 gh issue를 만든다. 중복 탐지 후, 크기를 추정해 label을 붙인다.

## Inputs

- **Feature name** — if omitted, read `.specs/CURRENT`.
- **--force** — 중복 issue가 있어도 새로 생성하고 싶을 때.
- **--label epic|task|sub-task** — 자동 추정을 무시하고 명시.

## Preconditions

1. `.specs/<feature>/manifest.yaml` 존재.
2. `.specs/<feature>/requirements.md` 존재 (issue body 재료).
3. `gh` CLI 사용 가능 + repo가 GitHub remote 보유.
4. `manifest.issue == null` (이미 있으면 한 줄 안내 후 stop — `--force`로 재생성 가능).

## Process

1. **Find feature** (인자 또는 CURRENT).

2. **Read manifest + requirements.md.**

3. **Search duplicates:**
   ```bash
   gh issue list --search "<feature> in:title" --state open --json number,title,url
   ```
   - 결과가 있으면 사용자에게 표 형태로 보여주고 선택:
     - a) 기존 issue 재사용 → manifest.issue 갱신, skill 종료
     - b) 새로 생성 (`--force`와 동일)
   - 결과 없으면 곧장 생성으로.

4. **Estimate label** (사용자가 `--label` 안 줬으면):
   - `.specs/<feature>/tasks.md`가 있으면 task 수로:
     - 1–4 → `sub-task`
     - 5–15 → `task`
     - 16+ → `epic`
   - tasks.md가 없으면 (sdd-new 전):
     - `.specs/<feature>/design.md`의 `## Affected modules` 항목 수로 근사
     - 모두 없으면 기본 `task`
   - `.specs/config.yaml`의 `issue.size_thresholds`로 override 가능

5. **Resolve assignee:**
   - `.specs/config.yaml`의 `issue.default_assignee` 사용
   - null이면 `gh api user --jq .login` (현재 user)

6. **Compose issue body:**
   ```markdown
   <requirements.md ## Background 한 단락>

   **Spec**: `.specs/<feature>/`

   ## Goals

   <requirements.md ## Goals>

   ## Acceptance criteria

   <requirements.md ## Acceptance criteria — 체크박스 그대로>

   ## Source

   <manifest.source 정보 — figma URL 또는 text 출처>

   ---

   🤖 Generated by sdd-plugin (sdd-to-issue). 진행 상태는 spec과 PR에서 추적됩니다.
   ```

7. **Compose title:**
   - 기본: requirements.md의 첫 h1 또는 feature name 기반
   - 예: `[task] add-search: 노트 인라인 검색`
   - prefix `[<label>]`를 붙여 한눈에 분류 보이게.

8. **Create issue:**
   ```bash
   gh issue create \
     --title "<title>" \
     --body-file <tmpfile> \
     --label "<label>" \
     --assignee "<assignee>"
   ```

9. **Parse returned URL** → issue 번호.

10. **Update manifest:**
    - `issue`: 번호
    - `issue_kind`: `epic` / `task` / `sub-task`
    - `parent_issue`: (이 skill에선 항상 null. sub-task가 부모를 가지려면 별도 skill 또는 수동.)
    - `last_skill: sdd-to-issue`
    - `updated_at: <today>`

11. **Report:**
    ```
    ✓ issue #142 created
      url:      https://github.com/.../issues/142
      label:    task
      assignee: hijack2ee

    manifest.issue: 142

    next: /sdd-branch
    ```

## Sub-task issues (epic인 경우)

이 skill은 단일 issue만 만든다. epic을 sub-task로 쪼개려면:

1. tasks.md를 sub-task가 될 단위로 쪼개기
2. 각 단위에 대해 별도 `.specs/<sub-feature>/`를 만들거나, 같은 feature 안에서 수동으로 `gh issue create --label sub-task` 실행 후 body에 `Tracked in #<epic>` 명시

자동 sub-task 생성은 향후 `--split` 옵션으로 추가 예정.

## Output format (duplicate found)

```
⚠ 비슷한 제목의 open issue가 있습니다:

  #138  [task] add-search: 노트 인라인 검색  (opened 2 days ago)
  #128  [epic] search-improvements           (opened 2 weeks ago)

옵션:
  a) #138 재사용 (manifest.issue: 138)
  b) #128 재사용
  c) 새로 생성 (--force)

어느 쪽으로?
```

## Output format (gh missing)

```
✗ gh CLI를 찾을 수 없습니다.

설치 후 인증: https://cli.github.com
혹은 issue를 수동으로 만들고 manifest.issue에 번호를 적어주세요.
```

## Constraints

- 중복 검색은 기본 동작. `--force`로만 우회.
- assignee는 항상 단일 (다중 할당은 별도 수동).
- label은 `epic`/`task`/`sub-task` 중 하나만. 다른 label은 손대지 말 것.
- 이미 만든 issue를 덮어쓰지 말 것. 갱신은 `gh issue edit` (이 skill 범위 밖).
- 한국어/영어 모두 OK — 사용자 컨벤션에 따름.

## When to use

✅ Use:
- /sdd-from-figma 또는 /sdd-from-text 직후
- spec이 있는데 issue가 빠진 경우 (수동으로 만든 spec)

❌ Don't use:
- 이미 manifest.issue가 있는 경우 (이미 issue 있음)
- 단순 질문/논의를 issue로 만들 때 (이 skill은 작업 issue용)
